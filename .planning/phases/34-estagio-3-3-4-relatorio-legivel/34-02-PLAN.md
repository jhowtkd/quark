# 34-02 — Estados de Erro e Vazio no Relatorio

## Goal
Garantir que o usuario sempre entenda o estado do relatorio: quando ele nao existe, quando uma secao falhou (com acao de retry), e qual a qualidade geral do relatorio gerado. Eliminar telas em branco ou estados de loading infinito sem feedback.

## Context
- `Step4Report.vue` possui um estado de espera (`waiting-placeholder`) que exibe animacao de loading quando `reportOutline` e null. Nao ha distincao entre "relatorio ainda nao iniciado", "relatorio nao encontrado" e "erro critico".
- As secoes do relatorio tem estados: loading (`currentSectionIndex === idx + 1`), completo (`generatedSections[idx + 1]`), e contaminado (`contaminatedSections`). Nao ha estado explicito de "secao falhou".
- O backend em `backend/app/api/report.py` retorna 404 quando relatorio nao existe, e o endpoint `/report/<report_id>/progress` retorna status `processing|completed|failed`.
- O `ReportAgent` no backend faz retry interno por secao (fase 1.2.2 do roadmap), mas o frontend nao tem visibilidade de falhas parciais nem botao de retry por secao.
- Nao existe nenhum indicador de qualidade do relatorio no frontend (ex: % de secoes completas, presenca de contaminacao, filtro de thinking process aplicado).

</action>

## Task 3: Estado de secao falha no frontend com retry

1. Adicionar estado `failedSections = ref(new Map())` (chave: sectionIndex, valor: `{error, retryCount}`).
2. Ao consumir o progresso do relatorio (polling ou via SSE), se o payload contiver `failed_sections`, popular `failedSections`.
3. No template, para cada secao em `failedSections`, renderizar um estado de erro dentro do corpo da secao:
   - `<div class="section-error-state">` com icone de alerta, mensagem de erro resumida, e botao "Tentar novamente".
   - O botao deve chamar funcao `retrySection(sectionIndex)`.

   - Em sucesso, recarregar o conteudo da secao e remover da `failedSections`.
   - Em erro, mostrar toast/snackbar com mensagem.
</action>

## Task 4: Endpoint de retry por secao no backend

<action>
Em `backend/app/api/report.py`:
1. Criar endpoint `POST /api/report/<report_id>/retry-section`:
   ```python
   @report_bp.route('/<report_id>/retry-section', methods=['POST'])
   def retry_section(report_id: str):
       data = request.get_json() or {}
       section_index = data.get('section_index')
       if not section_index:
           return jsonify({"success": False, "error": "section_index required"}), 400
       # Recuperar relatorio, verificar status, reexecutar apenas a secao falha
       ...
   ```
2. O endpoint deve:
   - Buscar o relatorio e verificar se existe.
   - Verificar se a secao realmente falhou (via `failed_sections`).
   - Reinstanciar `ReportAgent` com os mesmos parametros originais.
   - Chamar metodo interno `agent.generate_section(section_index)` (ou equivalente) em background thread.
   - Retornar imediatamente `{"success": true, "data": {"status": "retrying", "section_index": N}}`.
3. Atualizar `failed_sections` removendo a secao em retry e limpando o arquivo antigo se existir.
</action>

- Grep confirma validacao de `section_index` com retorno 400.
- Grep confirma retorno com `"status": "retrying"`.
</acceptance_criteria>

## Task 5: Indicador de qualidade do relatorio

1. Adicionar componente/area `ReportQualityIndicator` (pode ser inline no Step4Report ou componente separado) no topo do painel esquerdo, abaixo do header do relatorio.
2. O indicador deve mostrar:
   - **Score geral**: percentual calculado como `(completedSections / totalSections) * 100`, arredondado para inteiro.
   - **Badge de contaminacao**: se `contaminatedSections.size > 0`, mostrar badge amarelo/laranja com numero de secoes contaminadas.

3. O indicador deve ter classe CSS `report-quality-indicator`.
4. O score deve atualizar em tempo real conforme secoes sao completadas.
</action>
